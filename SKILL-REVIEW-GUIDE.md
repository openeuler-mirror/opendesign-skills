# SKILL-REVIEW-GUIDE.md

Skill 自评与进度管理的操作指南。每完成一个 Skill 后按此流程执行。

---

## 自评与迭代

每完成一个 Skill 后，必须进行自评。选择适用的评价维度（至少 6 项），逐维度审视并修正问题，发现即改不延迟。

### 评价维度（11 项，根据组件特点选择）

| 维度 | 关注点 |
|------|--------|
| **完整性** | Props、Events、Slots、Expose 是否全部覆盖？枚举值是否列全？ |
| **准确性** | 默认值、类型、条件逻辑是否与源码一致？v-model 识别是否正确？ |
| **层级关系** | 插槽嵌套/互斥关系是否正确反映了 template 结构？ |
| **响应式覆盖** | 是否体现了不同断点（≤840px / 841–1200px / 1201–1680px / >1680px）和交互形式（触控 vs 指针）下的行为差异？ |
| **场景覆盖** | 典型场景是否覆盖了 `__docs__/__case__/` 中的实际用法？推演是否有源码依据、无凭空假设？ |
| **可用性** | 设计理解卡（Part A）是否真的不含代码术语、类型签名？非开发人员能否快速理解组件能做什么？ |
| **可拼装性** | 代码模板能否直接复制使用？Props、Events、Slots 名拼写是否正确？示例中的 prop 值是否都在可选值范围内？ |
| **一致性** | Part A（设计理解卡）与 Part B（代码调用参考）之间是否对齐？有无矛盾或遗漏？ |
| **LLM 友好度** | 大模型能否仅凭此 Skill 正确使用该组件？检查：①信息自包含；②无歧义表述；③代码模板完整可直接组装；④约束和互斥关系显式声明 |
| **布局准确性** | 布局结构图是否与 Playwright 快照吻合？slot/prop 标注位置是否与实际视觉位置一致？多断点布局差异是否体现？ |
| **Pixso MCP 可用性** | 设计师在 Pixso 中看到的视觉结构，能否通过布局结构图和设计理解卡快速映射到组件 API？设计工具术语使用是否恰当？ |

### 自评流程

1. **回溯源码验证**：不凭记忆，逐维度回溯源码验证关键事实（如 Props 列表、Events、Slots 等）
2. **发现问题立即修正**：在 Skill 内容中直接修正，不只是记录问题
3. **输出自评报告**：将自评结论和源码修正清单写入 `{name}.review.md`（**不放在 Skill 文件内**）

### 自评报告格式（写入 `{name}.review.md`）

```markdown
#### 源码修正清单（如有修正则必填）

| 文件 | 修正类型 | 修正内容 | 原因 |
|------|---------|---------|------|
| types.ts | JSDoc 更正 | `disabled` prop 注释从"是否禁用按钮"改为"是否禁用表单提交" | 与实际行为一致 |

#### 自评结论

| 维度 | 结果 | 备注 |
|------|------|------|
| 完整性 | ✅ 通过 | 46 个 props 全覆盖，7 个 events 全覆盖，5 个 slots 全覆盖 |
| 准确性 | ⚠️ 已修正 | `size` prop 默认值原写为 'medium'，实为 'm'，已更正 |
| 布局准确性 | ✅ 通过 | ASCII 布局图与 4 个断点快照完全吻合 |
| ... | | |

整体判定：✅ 可交付 / ⚠️ 有保留（说明原因）
```

## 进度文件管理

**重要**：每完成一个 Skill 后，**必须立即更新进度文件**，确保中断后可恢复。

进度文件的位置和命名：
- 组件 Skill：`skills/opendesign-components/references/_skill-gen-status.md`
- 脚本 Skill：`skills/opendesign-scripts/references/_skill-gen-status.md`

进度文件格式：
```markdown
# Skill 生成进度

## 已完成（N/46）
- [x] Anchor
- [x] Badge
...

## 待处理（M/46）
- [ ] Button
- [ ] Card
...

## 当前状态
- **最后更新**：2026-05-15
- **当前任务**：Button 组件
- **源码问题**：Button.vue 中 defineEmits 声明不完整，待补充 3 个事件
```
