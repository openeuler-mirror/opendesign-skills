# SKILL-GEN-GUIDE.md

Skill 生成的详细操作指南。适用场景：新建或更新组件/脚本 Skill 时按需查阅。

---

## 核心工作流

### 1. 生成组件 Skill

当需要生成或更新某个组件的 Skill 文件时：

**单个组件**：
```bash
# Agent 会根据指定的组件名（PascalCase），从 @opensig/opendesign 源码中提取 API，生成 Skill
# 例如：生成 Card 组件的 Skill
```

**批量生成**：
```bash
# 扫描组件库，遍历所有含 types.ts 的子目录，逐个生成 Skill
# 按目录字母序处理，每完成一个就更新进度文件
```

**关键点**：
- Skill 文件写入 `skills/opendesign-components/references/{component}.md`
- 每完成一个组件，**必须更新** `skills/opendesign-components/references/_skill-gen-status.md`（记录已完成和待处理列表）
- 自评结论写入 `{component}.review.md`，**不要混入 Skill 文件本身**
- 通过 Playwright 截图进行视觉布局分析（如快照不存在，负责生成）

#### 详细的组件 Skill 生成流程（7 个步骤）

**前置：确认源码版本**

在提取任何 API 之前，先到组件库源码目录（通常为本项目同级的 `opendesign/` 或 `packages/` 下）查看当前版本：

```bash
git describe --tags --abbrev=0   # 最近发布的版本 tag
# 只追踪 types.ts 中目标接口定义的历史，避免读取整个文件
git log -L '/ButtonProps/,/^}/' -- packages/components/button/types.ts
```

`git log -L` 只输出匹配行块的变更历史，优先使用。看不出上下文时再用 `git show <hash>` 查具体 commit，仍不够时才读完整文件——梯次决策，按需升级。

- 将版本号记录在 Skill 文件头部（格式见下方「Skill 文件格式规范」）
- 若更新已有 Skill，对比上次记录的版本与当前版本间的接口变更（见下方「更新文档」）
- 破坏性变更的判断依据是 commit message（`feat!`、`BREAKING CHANGE`）和接口定义行块的 diff

1. **提取 Props**：读取 `types.ts` 中的 `{Name}Props` 对象，提取字段名、类型、是否必填、默认值、中文注释
2. **提取 Events**：从 `O{Name}.vue` 的 `defineEmits` 提取事件名、参数类型、触发时机；识别 v-model 支持
3. **提取 Slots**：结合 `defineSlots` 声明和 `<template>` 中的 `<slot>` 标签，提取插槽名、slot props、渲染条件、回退内容
4. **提取 Expose**：检查 `defineExpose` 声明，提取暴露的公开方法及其签名
5. **分析响应式差异**：检查 `style/media.scss`（CSS 变量覆盖）和 `useScreen()` 调用（运行时适配），识别不同断点和交互形式下的行为差异
6. **学习真实用法**：阅读 `__docs__/__case__/` 下的示例文件，提取实际用法模式、使用场景、隐含约束，举一反三推演合理的用法组合
7. **视觉布局分析**：获取 Playwright 快照，对照代码与视觉快照识别布局结构、区域划分、对齐方式、层叠关系，绘制 ASCII 布局图

### Playwright 快照基础设施

Skill 中的布局结构图需要以 Playwright 快照为视觉依据。使用快照前应确保其可用：

#### 快照检查与生成

1. **检查现状**：查看 docs 子包（`packages/docs`）中是否已有 `__snapshots__/{name}/` 目录及对应的 PNG 快照文件
2. **视口标准**：快照应覆盖以下 4 个断点（与响应式设计对应）：
   - `phone`（≤600px）：375×812
   - `phonePad`（601–840px）：1024×768
   - `laptop`（1201–1680px）：1440×900
   - `widescreen`（>1680px）：1920×1080
3. **生成快照**（如缺失）：
   - 启动 docs 开发服务器（`cd packages/docs && npm run dev`）
   - 定位组件文档页面 URL（通常为 `/components/{name}`）
   - 使用 Playwright 对各断点的 demo 逐一截图，保存到 `__snapshots__/{name}/demo-{index}-{viewport}.png`
   - 执行：`cd packages/docs && npx playwright test __snapshots__/{name}/ --project=phone --project=phonePad --project=laptop --project=widescreen`

#### 布局分析要点

- **收集快照**：获取 4 个断点的完整快照集
- **对比变化**：将不同视口的快照逐一对比，记录布局方向切换、区域合并、元素尺寸/间距变化
- **验证代码**：将快照中的视觉结构映射回 `<template>` 代码，确保 slot/prop 标注位置准确
- **绘制模型**：用 ASCII 图描述各断点的布局，标注 slot 名、布局方向、对齐方式

### 2. 生成脚本 Skill

当需要生成 CLI 命令的 Skill 时：

**单个命令**：
```bash
# 从 @opensig/open-scripts 的源码中提取命令定义和配置格式，生成 Skill
```

**批量生成**：
```bash
# 遍历所有注册的命令，逐个生成 Skill，每完成一个更新进度文件
```

**关键点**：
- Skill 文件写入 `skills/opendesign-scripts/references/{command}.md`
- 需要在 monorepo 中查找该命令的真实使用配置（如 `icons/icon.config.ts`）
- 自评结论写入 `{command}.review.md`

#### 详细的脚本 Skill 生成流程（3 个步骤）

1. **提取命令元信息**：从 CLI 入口文件 `src/index.ts` 的 Commander 注册中提取命令名、描述、选项（标志、说明、是否必填）
2. **提取配置字段**：从 `src/{command-kebab}/config.ts` 或 `index.ts` 中提取配置接口定义，对每个字段提取名称、TypeScript 类型、是否必填、默认值、用途说明；展开嵌套类型的子字段
3. **收集真实示例**：在 monorepo 中搜索该命令的实际调用，读取对应的配置文件，记录来源、配置差异（不同子包间的差异）、`package.json scripts` 的集成方式

#### 脚本 Skill 文件的输出格式

**Part A：功能简介**

一段简短的自然语言描述，说明这个命令**做什么、什么时候用**，包括：
- 具体例子（而非抽象说法）
- 与其他命令的执行顺序关系（如"通常在 `gen:icon` 之后执行"）
- 前置依赖说明（如"需要先执行 X 命令"）

**Part B：调用与配置参考**

```
#### 命令用法
open-scripts {cmd} [options]

| 选项 | 说明 | 必填 | 默认值 |
|------|------|------|--------|
| -c, --config <path> | 配置文件路径 | 否 | ./xxx.config.ts |

#### 配置文件格式
| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|

#### 配置示例
展示 2–3 个来自 monorepo 的真实配置文件，标注来源

#### package.json 集成
展示脚本集成的推荐方式

#### 与其他命令的执行顺序
前置命令 / 后续命令表

#### 注意事项
关键使用要点、边界情况
```

### 3. 更新文档

当组件库或工具发生变化时，更新相应的 Skill 文件：

```bash
# 按文件列表逐个修改，使用 Edit 工具进行增量修改（而非重写整个文件）
# 修改完成后，验证 Markdown 格式是否正确
```

**更新时的版本比对流程**：
1. 查看 Skill 文件头部记录的旧版本号
2. 在组件库源码目录，按需逐步放宽查询范围：

   **第一步：只看接口行块**（多数情况够用）
   ```bash
   git log <旧版本tag>..HEAD -L '/{Name}Props/,/^}/' -- packages/components/{name}/types.ts
   ```

   **第二步：行块历史看不出上下文时，查具体 commit 的完整 diff**
   ```bash
   git show <commit-hash> -- packages/components/{name}/types.ts
   ```

   **第三步：变更涉及模板或运行时逻辑时，读整个文件当前状态**
   ```bash
   # 用 Read 工具读 O{Name}.vue，对照行块历史判断影响范围
   ```

   按需升级，能在第一步确认的不进入第二步。

3. 有破坏性变更 → 更新「版本迁移说明」节；无破坏性变更 → 只更新头部版本号即可

## Skill 文件格式规范

### 组件 Skill（以 Card.md 为例）

**版本标注**
- 在文件头部导航链接之后，用一行注明 Skill 对应的源码版本：
  ```
  > 本文档对应 @opensig/opendesign **v2.3.1**（2025-11 生成）
  ```
- 若该组件曾有过 API 破坏性变更，在文件末尾增加「版本迁移说明」小节，列出各版本间需要手动调整的地方（重命名的 prop、移除的 slot、变更的事件签名等）；无破坏性变更则省略此节

**Part A：设计理解卡**
- 目标读者：设计师或不熟悉代码的人
- 用自然语言描述组件的视觉行为和交互
- 必须包含：**布局结构**（ASCII 图）和**响应式行为**（如有差异）
- 不出现代码、类型签名等开发术语

**Part B：代码调用参考**
- Props 表、Events 表、Slots 表
- Slot 层级关系图（嵌套/互斥关系）
- 组件布局结构图（多断点版本）
- 典型使用场景与代码模板
- 响应式行为表（6 列，高亮 ★ 关键阈值）

### 脚本 Skill（以 gen-icon.md 为例）

**Part A：功能简介**
- 一段自然语言描述：这个命令做什么、什么时候用
- 说明前置依赖和后续命令

**Part B：调用与配置参考**
- 命令用法（CLI 选项表）
- 配置文件格式（字段表）
- 真实配置示例（从 monorepo 中摘取）
- package.json 集成示例
- 与其他命令的执行顺序

### 导航链接（所有 reference 文件）

所有 reference 文件头部均包含返回链接：

```
> ← [组件索引](../SKILL.md#组件索引) · [README](../../../README.md)
```

## 响应式断点生成要求

Skill 中的**响应式行为表必须覆盖 4 个断点**（≤600px / 601–840px / 841–1200px / >1680px），对应 opendesign 库的三个关键阈值（**★ 标注 840px / 1200px / 1680px**）。

**完整的断点定义、CSS 变量规则、触控设备差异见**本文件末尾「[全局约定](#全局约定生成时必须遵守)」章节。

生成 Skill 时的关键要求：
- 从 `style/media.scss` 提取各断点的 CSS 变量覆盖值
- 从 `useScreen()` 调用识别运行时响应式差异（`isPhonePad` / `isPhonePadSize` / `isTouchDevice`）
- 在响应式行为表中用 6 列格式清晰展示多断点布局和交互变化
- 标注关键阈值（★），帮助 AI 根据设计稿尺寸选择对应断点

**响应式行为表标准格式**（6 列，高亮关键阈值）：

| 维度 | ≤600px | **601–840px** ★ | **841–1200px** ★ | 1201–1680px ★ | >1680px |
|------|--------|----------------|-----------------|--------------|--------|
| （示例行） | — | — | — | — | — |

> 说明：
> - `≤600px`（纯手机）与 `601–840px`（平板竖屏）通常行为相同，opendesign 以 840px 为第一阈值
> - `1201–1680px` 为标准桌面，`>1680px` 为宽屏——部分组件在宽屏下有专属布局变化
> - `★` 标注的三列是组件行为发生实质变化的关键阈值
> - 若某行在某列无变化，填 `同前` 或重复当前值，不留空

## Pixso MCP 协同指南

生成的 Skill 需要支持 Pixso MCP 工作流，让 AI 能从设计稿中识别组件并生成代码。

### Skill 中的信息如何支持设计稿匹配

| Skill 中的信息 | 匹配作用 |
|---------------|---------|
| **布局结构图** | AI 将设计稿的图层结构与布局结构图对比，判断是否匹配该组件 |
| **设计理解卡 → 属性描述** | AI 根据设计稿的视觉表现（尺寸、颜色、变体）推断对应的 prop 值 |
| **布局结构图 → slot 标注** | AI 识别设计稿中超出标准属性的自定义区域，映射为 slot 用法 |
| **响应式行为表** | AI 根据设计稿的画板尺寸，选择对应断点的布局和交互方式 |

### 设计理解卡（Part A）的写作要求

为支持 Pixso MCP 协同，Part A 需遵循额外规范：

#### 1. 使用设计工具术语
在描述组件时使用 Pixso/Figma 设计师熟悉的概念，帮助 AI 建立设计语言与代码的映射：

| 设计概念 | 对应说明 |
|---------|---------|
| "自动布局" | 指 flex 布局，内容自动排列且尺寸自适应 |
| "固定宽度"、"固定高度" | 指组件或区域的尺寸被锁定，不随内容变化 |
| "填充容器" | 指元素宽度/高度铺满父容器 |
| "绝对定位浮层" | 指 Popup、Overlay 等浮于页面之上的层 |
| "两端对齐"、"居中对齐"、"左对齐" | 指 flex 的 justify-content 对齐方式 |
| "紧凑间距"、"宽松间距" | 指元素间的相对间距（相对于组件尺寸） |

#### 2. 标注视觉层级
明确区分哪些是"容器""内容""装饰"，与设计稿图层结构对应：
> - **容器区域**（背景、边框）：用 `background` 和 `border` 属性控制
> - **内容区域**（文字、图标）：通过插槽传入，占据主要视觉空间
> - **装饰元素**（分割线、徽标）：通过 prop 控制显隐

#### 3. 描述间距与尺寸规律
写入具体的间距和尺寸信息，与设计稿中的约束对应：
> 组件内部四周有 16px 内边距；标题与描述间距 8px；图标与文字间距 12px。在平板尺寸下，内边距缩减为 12px。

#### 4. 区分固定区域与弹性区域
说明哪些区域尺寸固定、哪些自适应内容：
> 顶部操作栏高度固定为 44px；中部内容区域自适应高度；底部按钮区域宽度固定为 320px（在移动端铺满屏幕）。

### 设计 Token 关联

如组件的样式中使用了设计 Token（CSS 变量），在设计理解卡中关联到 Pixso 侧的设计变量名称：

> 组件主背景色使用 `--o-color-bg-card`（openEuler 主题对应 `#f5f5f5`，Ascend 主题对应 `#ffffff`），对应 Pixso 中的 `color/bg/card` 变量。

## 源码修正规则

在生成或更新 Skill 前，若发现源码中存在以下问题，**应该直接修正源码**（而非绕过）：

| 问题类型 | 修正范围 |
|---------|---------|
| JSDoc 缺失或不准确 | 补充/更正 `@zh-CN` 注释 |
| Props 默认值与代码不一致 | 以实际代码为准更正类型定义 |
| defineSlots 与 template 不一致 | 以 template 为准更正声明 |
| case 文件缺少关键示例 | 新增相应的场景示例文件 |
| 示例代码错误 | 直接修正 |

**修正原则**：
- 只修正**明确错误和关键缺漏**，不做风格调整
- 新增 case 文件命名遵循现有约定
- 所有修正记录在 `{name}.review.md` 的「源码修正清单」中

## 全局约定（生成时必须遵守）

各 skill 文件只记录**与以下约定不同的例外情况**，无需重复说明。

### 组件命名

- 所有组件均以 `O` 前缀命名（如 `OButton`、`OInput`）
- 从 `@opensig/opendesign` 统一导入

### 响应式断点定义

opendesign 生态涉及两套断点体系，使用时注意区分：

**opendesign 组件库断点（isPhonePad 体系）**

组件库运行时通过 `useScreen()` 检测，三个关键阈值：

| 条件 | 阈值 | 含义 |
|------|------|------|
| `isPhonePadSize` | **≤840px** ★ | 触控设备竖屏（手机 + 平板竖屏），组件布局发生主要变化 |
| `isPhonePad` | **≤1200px** ★ | 所有触控/平板设备，UI 切换为紧凑模式 |
| 宽屏阈值 | **>1680px** ★ | 大屏/宽屏，部分组件有额外的宽屏布局 |

> ★ 840 / 1200 / 1680 是核心边界。

**openEuler Portal 项目断点（screen.scss 体系）**

Portal 项目通过 `@include respond-to('xxx')` 使用：

| 名称 | 范围 | 说明 |
|------|------|------|
| `phone` | 0–600px | 手机 |
| `pad_v` | 601–840px | 平板竖屏 |
| `pad_h` | 841–1200px | 平板横屏 |
| `pad` | 601–1200px | 全部平板（pad_v + pad_h） |
| `laptop` | 1201–1440px | 笔记本 |
| `<=pad` | 0–1200px | 平板及以下 |
| `<=laptop` | 0–1440px | 笔记本及以下 |
| `>pad` | ≥1201px | 桌面及以上 |
| `pad-laptop` | 601–1440px | 平板 + 笔记本 |

### Slot 渲染约定

- slot 未传入时，组件**不渲染**对应区域（除非该 skill 文件另有说明）
- 外层 slot 被使用时，内部所有子 slot **全部失效**（另有说明除外）

### 触控 vs 指针设备

通过 `isTouchDevice` 检测：

| 设备类型 | 说明 | 影响 |
|---------|------|------|
| 触控设备（`isTouchDevice=true`） | 手机、平板 | hover 交互退化为 click；部分弹出层行为改变 |
| 指针设备（`isTouchDevice=false`） | 桌面鼠标 | 完整 hover 效果；tooltip 等 hover 触发组件正常工作 |

### 组件 CSS 变量覆盖

所有组件均支持在调用处直接覆盖 CSS 变量，**无需 `:deep` hack**：

```vue
<OButton style="--btn-min-width: 120px" />
```

各组件的可覆盖变量列表见各 skill 文件的「可覆盖的 CSS 变量」章节。
